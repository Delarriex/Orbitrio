# Orbitrio Migration Notes

**Migration complete.** This project has fully migrated off Firebase (Auth + Firestore + Storage) onto **Clerk** (auth) and **Supabase** (database + storage). Firebase has been deleted from the codebase — no `firebase` package, no `lib/firebase.ts`, no `.rules`/`.json` config files, zero remaining imports. This doc exists so any future work — by Claude Code, another AI tool, or a human — can pick up without re-deriving the whole history, and remains useful as a record of *why* things are shaped the way they are (RLS patterns, RPC atomicity rules, known quirks preserved from the old system).

## High-level architecture

- **Auth**: Clerk. Fully migrated. `AuthPage.tsx` uses Clerk's `useSignUp`/`useSignIn` hooks directly (not prebuilt Clerk components, to preserve custom styling).
- **Identity/role**: `src/hooks/useCurrentUser.ts` — combines Clerk's `isSignedIn` with a Supabase `users.role` lookup. This is the single source of truth for "is this user an admin" used in routing guards (`App.tsx`, `DashboardAdmin.tsx`).
- **Data layer pattern**: each migrated feature gets its own hook in `src/hooks/data/`, following a consistent shape: fetch on mount, expose CRUD functions, use Supabase RPC functions for anything that touches balance (atomicity). `OrbitContext.tsx` calls these hooks and exposes their data/functions through `useOrbit()`, so consuming components didn't need to change.
- **Supabase client**: `src/lib/supabase.ts` — `useSupabaseClient()` is memoized on `session?.id`. **This memoization is critical** — an earlier bug where the client was recreated every render caused an infinite update loop across every data hook. Do not remove the `useMemo`.

## Still on Firebase / not yet migrated

Nothing. Firebase has been fully deleted — see "Firebase deletion — complete" below for exactly what was removed.

### File storage — now on Supabase Storage

Two buckets, both created by `storage_buckets.sql`:

- `deposit-proofs` (private) — deposit proof-of-payment uploads from `DashboardWallet.tsx`, via `uploadDepositProof()`. Objects live under `{clerkUserId}/...`; RLS policies on `storage.objects` restrict insert to your own folder and select to owner-or-admin. **Stores the object path, not a public URL** — deliberately different from the old Firebase Storage `getDownloadURL()` behavior, which issued effectively-public bearer-token links. This is safe because `AdminDepositsTab.tsx` never actually rendered `proofFile` as a clickable link/image (just a static "Proof attached" label if truthy) — nothing regresses, and a private bucket is more appropriate for financial documents. A future "view proof" admin feature would need to generate a signed URL from the stored path.
- `deposit-wallet-qr` (public read, admin-only write) — deposit-wallet QR code images from `AdminWalletsTab.tsx`, via `uploadDepositWalletQrCode()`. These **are** rendered directly in `<img src>` tags (`AdminWalletsTab.tsx`, `GlobalModals.tsx`), so the bucket must stay public and the helper returns a real public URL via `getPublicUrl()`.
- Checked `DashboardKYC.tsx` for a third upload flow (implied by `storage.rules`' now-vestigial `kyc/` path) — found none. No `storage`/`uploadBytes`/`FileReader` anywhere in that file; KYC image fields are never actually populated via Storage today. Scope stayed at exactly these two flows.

### Airdrop claims — now on Supabase

New `airdrop_claims` table (distinct from the already-migrated `airdrops` campaign catalog). Design notes:

- Submission is a plain RLS-gated insert (no balance touched) — same reasoning as support ticket creation. Approve/reject are `security definer` RPCs, since approval credits the claimant's balance.
- `approve_airdrop_claim` re-checks `status = 'Pending'` under a `select ... for update` row lock before crediting, replacing the old Firestore version's `payoutTransactionId`-presence dedupe check with a stronger row-level lock against a double-click/race.
- RLS on `select` is the same per-user-or-admin pattern as everything else (`user_id = auth.jwt()->>'sub' OR is_admin()`) — this exactly replicates the old Firestore listener's behavior (non-admins got a query filtered to their own `userEmail`; admins got the whole collection). **Preserved a pre-existing quirk rather than fixing it**: because non-admins only ever see their own claims, `DashboardAirdrops.tsx`'s claim-limit progress bar and "limit reached" gate (both driven by `getCampaignClaimCount`/`hasReachedClaimLimit` over the same claims list) have likely never worked correctly for regular users — their own claim count against a campaign is always 0 or 1, never near a real limit like 50. This was already broken before the migration; fixing it would mean exposing all users' claims to all users, a bigger behavior change than a straight migration should make unilaterally.
- The `transactions` row inserted on approval uses `asset: 'USD'` (not the claimed token), matching the original `buildTransaction("tx-airdrop", "payout", reward, "USD", ...)` call exactly.
- `adminAirdropClaims` was the last real-world consumer of the Firestore-sourced `adminUsers` state — see note above.

### Notifications — now on Supabase

`useNotifications.ts` is shaped differently from every other feature hook: it owns **no React state**, just five thin async functions (`fetchNotifications`, `saveNotificationToDb`, `markReadInDb`, `markManyReadInDb`, `deleteNotificationInDb`) — a direct swap-in for what `notificationsService.ts`'s Firestore functions used to do. `OrbitContext.tsx` still owns the `notifications` array and all its business logic (`addNotification`, `syncNotificationLocally`, `shouldShowNotificationLocally`, mark-read/delete with optimistic local updates) exactly as before, unchanged line-for-line apart from which backend function each calls. Reason: every other feature is a straightforward "hook owns the canonical list, `OrbitContext` re-exports or overlays it via one sync effect," but notifications already had **optimistic local mutations layered directly on top of a live subscription within one `useState`** — introducing a second, separately-fetched state here would create two sources of truth that could clobber each other (e.g. a sync-effect re-running with stale data right after an optimistic mark-read).

Other notes:
- **No live push** — the old Firestore `onSnapshot` gave instant cross-user delivery (e.g. an admin's approval notification appearing in the user's bell without a refresh); Supabase here is fetch-on-mount only, same as every other migrated feature (bug #9's pattern). A notification for a different user's session shows up on their next natural refetch, not instantly. Not fixed here — deliberately deprioritized as "degrades silently" rather than "throws," per the reprioritization after bug #11. Realtime/polling would be new infrastructure and wasn't requested.
- **Two small deliberate deviations from the old Firestore rules**, both loosening rather than tightening: the recipient can now delete/fully-update their own notifications (old rules restricted delete to admins only and update to flipping `read` only) — the app's own "Clear"/dismiss UI clearly intends for a user to manage their own notifications, so this closes what looked like a rules oversight rather than replicating it.
- `notificationsService.ts` kept its pure, Firebase-free functions (`buildNotification`, `normalizeNotification`, `sortNotifications`, `formatRelativeTimestamp`, `getNotificationDocumentId`) — only the five Firestore-touching functions and the `lib/firebase` import were removed.
- A `notifications` table already existed in the live Supabase project before this step, with an unknown/different origin — see the SQL file list above.

### Investment plan catalog — now on Supabase

New `investment_plans` table — the admin-editable Bronze/Silver/Gold/Platinum/Diamond tier list (ROI, min/max deposit, duration, enabled/paused). **Distinct from "Active investments" (already migrated earlier) — that's users' own purchased positions; this is the catalog they invest into.** Smaller scope than expected once investigated: the actual money-moving side (`investInPlan`/`claimPlanPayout`/`topUpInvestment` → `useActiveInvestments.ts` → `create_investment`/`claim_investment_payout`/`top_up_investment` RPCs) was already fully on Supabase — only the plan-template CRUD itself was still Firestore.

Design notes:
- Plain CRUD, no RPCs — matches `deposit_wallets`' shape exactly (`useInvestmentPlans.ts` is modeled directly on `useDepositWallets.ts`). Public read (`using (true)`), admin-only write, mirroring the old Firestore rule (`allow read: if true; allow write: if isAdmin();`) exactly.
- Admin write (create/update/delete/toggle) had been **silently broken** since the Clerk cutover — Firestore's `isAdmin()` requires `request.auth != null`, which is never true (bug #11's pattern). Public read still worked (`allow read: if true`), so the catalog itself displayed fine; only admin management was dead.
- The old client-side "seed 5 defaults if the collection is empty" logic (gated on `auth.currentUser?.email` being an admin email) was also dead for the same reason, so it never actually seeded anything in production. Replaced with a one-time idempotent seed directly in `investment_plan_functions.sql` (`on conflict (id) do nothing`).
- `investmentsService.ts` kept its pure functions (`normalizeInvestmentPlan`, `sortInvestmentPlans`, `getEnabledInvestmentPlans`, `buildActiveInvestment`, etc.) — only the six Firestore-touching functions (`watchInvestmentPlans`, `seedDefaultInvestmentPlans`, `saveInvestmentPlan`, `createInvestmentPlan`, `deleteInvestmentPlan`, `setInvestmentPlanEnabled`) and the `lib/firebase` import were removed, along with the now-unused `INVESTMENT_PLANS_COLLECTION` export.

### KYC Firestore fallback — removed (not migrated, deleted)

Unlike every other round, this wasn't a migration — investigation showed the fallback path was **already dead in production**, so it was deleted outright rather than ported to Supabase. Two things were removed, both rooted in the same dead `users/{email}` Firestore path (bug #11):

- `kycService.ts`'s four Firestore functions (`createSubmission`, `updateSubmission`, `getUserSubmission`, `adminReview`) and their call sites in `OrbitContext.tsx` (`submitKyc`'s `else` branch, `adminKycReview`'s `else` branch). Both branches only fired when the primary Supabase path had no user ID/row to work with — but `ensureUserRow(supabase, clerkUser)` runs unconditionally on every login before any UI is interactive, so every real signed-in user already has a Supabase row by the time they could trigger either function. The fallback was unreachable, and even if reached, would have thrown a permission error (the same `isAuthenticated()` rule bug #11 already covers). `kycService.ts` now only exports `KYC_DOCUMENT_TYPES` (still used by `DashboardKYC.tsx`'s document-type dropdown).
- The `legacy` Firestore bridge in the profile-loader effect (`OrbitContext.tsx`, the effect that runs on every login) — a `getDoc(doc(db, "users", email))` call that seeded `user.balance`/`portfolioValue`/`activeInvestments`/`copyTrades`/`portfolio`/`transactions`/`tickets` from a stale "not yet migrated" comment. All seven fields have had their own real Supabase-sourced sync effect since earlier rounds (`balance` ← `currentUserProfile.balance`, etc.), so this `getDoc` call already always failed with a permission error (same rule) and was silently caught, falling back to the exact same zero/empty defaults now hardcoded directly in the initial `setUser({...})` call. **Removing it is provably behavior-neutral** — the Firestore read has had a 100% failure rate since the Clerk cutover, so today's runtime behavior (zero/empty until the sync effects overlay real data a moment later) is unchanged; this just removes the wasted network round-trip and the dead code pretending to do something.

### Firebase deletion — complete

Final round. Turned out to be lower blast-radius than the "highest-risk step" label implied: only `OrbitContext.tsx` actually imported from `lib/firebase.ts` anywhere in `src/` (confirmed by a repo-wide grep before touching anything). Deleted, in order:

**Dead code removed from `OrbitContext.tsx`:**
- `adminUsers` state, `INITIAL_MOCK_USERS`, `SimulatedUser`-typed plumbing, and the `onSnapshot(collection(db,"users"))` listener effect that populated it — zero real-world consumers (confirmed: only referenced inside the already-dead mock `login()` and the mock-only `markAnnouncementRead` sync, both also removed).

Since Firebase Auth is dead (bug #3), this guard never passed; the entire effect body (a `setTimeout`-debounced `setDoc` writing `balance`/`transactions`/`portfolio`/etc. to `users/{email}`) had been unreachable since the Clerk cutover. Only its `localStorage.setItem("orbitrio_user", ...)` line at the top was live — kept as its own minimal effect.
- `handleLog`'s Firestore write to `audit_logs` — checked, zero reads of that collection anywhere; the real audit-log source is local state + `localStorage`. Pure dead write, removed; local logging untouched.
- Now-orphaned pure helpers/imports: `normalizeLedgerTransactions`, `buildRegistrationUserDoc`, `buildSimulatedUserFromRegistration`, `sendSecurityAlert` destructure (all only referenced from the deleted functions above).

**Live bugs fixed as a side effect of the same deletions** (not just cleanup — worth testing explicitly): three functions had an *unguarded* Firestore call that threw and aborted the function before reaching their already-correct Supabase logic a few lines later. Deleting the dead Firestore line unblocks the working code right after it:

- `saveWalletConnection` — the wallet-connection preference never actually persisted to Supabase.
- `markAnnouncementRead` — the `user_read_announcements` upsert never ran.

See bug #13 below for the general pattern this represents.

**Files/config deleted**: `src/lib/firebase.ts`, `firestore.rules`, `storage.rules`, `firebase.json`, `firestore.indexes.json`, `firebase-applet-config.json`, `firebase-blueprint.json`, the `firebase` npm package (`npm uninstall firebase`), the `VITE_FIREBASE_*`/`VITE_FIRESTORE_DATABASE_ID` block from `.env.example` (real `.env` untouched), and the now-stale `firebase-applet-config.json` entry in `.gitignore`.

**Kept as pure/general-purpose (not Firebase-specific, still used)**: `KYC_DOCUMENT_TYPES` in `kycService.ts`; `enrichTransaction`, `isAdminEmail`, and other pure builder/normalizer functions throughout `services/`; `sendSecurityAlert`'s definition in `emailClient.ts`/`useEmailNotifications.ts` (only its one dead call site was removed — the function itself is general email infra, left alone as it wasn't in scope).

Verification: `npm run lint` (`tsc --noEmit`) clean, `npm run build` (Vite + esbuild server bundle) succeeded, repo-wide case-insensitive grep for `firebase` returns only comments/this doc/SQL-file comments — zero remaining code references.

## Routing — now react-router-dom (post-migration UI/UX phase)

The original hand-rolled routing (`currentView` state + manual `pushState`/`popstate` listeners in `App.tsx`, called out as technical debt from the start) was replaced with real react-router-dom v7 routes + guards. Key facts for future work:

- **Why**: refreshing while signed in flashed the public home page (currentView initialized to `"home"`, corrected only after Clerk + the Supabase role resolved). Admins double-flashed (home → user dashboard → admin) because `isAdmin` resolves later than `isSignedIn`.
- **The fix that matters**: `AppShell` renders a full-screen spinner until `useCurrentUser().isReady` (`clerkLoaded && !profileLoading`) — routes never mount before identity + role are known. If a "wrong page flashes on load" bug ever reappears, check this gate first.
- **Adapter pattern**: components throughout the app still call `onNavigate("dashboard-wallet")` etc. — `VIEW_TO_PATH` in `App.tsx` maps view strings → URLs, `PATH_TO_VIEW` maps back for active-link highlighting. Adding a page = add a route + both map entries; don't reintroduce `setCurrentView`.
- Trading asset / wallet sub-tab moved into query params (`/dashboard/trading?asset=BTC`, `/dashboard/wallet?tab=support`) — these now survive refresh (they didn't before). Route wrappers (`TradingRoute`/`WalletRoute`) key the page component on the param so re-navigation remounts it.
- Legacy hash deep-links (`/#markets`, `/#/admin`) redirect to real paths via `LEGACY_HASH_TO_PATH`. `/#about-us` and `/#contact` are genuine home-page scroll anchors (a shell effect scrolls with retries while the lazy chunk loads — this also fixed About/Contact not scrolling when clicked from another page).
- Admins are pinned to `/admin` from any path (pre-router behavior, preserved). `AdminOverviewTab` still sets `window.location.hash` directly for admin sub-tabs — outside the router but harmless since route matching ignores the hash.
- The "TEMP: Clerk sign-in test" box (white box, top-right) was removed at the same time.
- Express (`server.ts`) already had a catch-all `index.html` fallback, so deep links needed no server change.

## Important bugs found and fixed during this migration (useful context, not just history)

1. **Supabase client not memoized** → infinite re-render loop across every data hook. Fixed in `lib/supabase.ts` — `useMemo(..., [session?.id])`. If you ever see "Maximum update depth exceeded" again, check this first.
2. **Write/read split** — several features had their write functions correctly pointed at Supabase, but the `user` object the UI actually displays was still populated from a legacy Firestore fetch in the profile-loader effect. Fixed via `useEffect` sync blocks (`setUser(prev => ({...prev, X}))`) that overlay each migrated field once its hook's data changes. **When migrating a new feature, always add one of these sync effects** or the UI won't reflect the new data.
3. **Silent Firestore write failures** — several old write paths (`copyTrades`, portfolio via a generic "sync everything" effect) were guarded by `if (!firebaseUser) return` — checking **Firebase Auth's own signed-in user**, which is always `null` now that auth runs through Clerk. These writes had been silently doing nothing since the Clerk migration (Step 3 of this project), not something introduced later. **If a feature seems to "not persist," check for this pattern first** — a Firebase Auth guard clause that can never pass.
4. **RLS + client-declared status** — the `transactions` table's RLS policy only allows users to insert their own transactions as `status = 'pending'`. A few flows (e.g. instant-complete deposits) needed to always insert as pending then flip to completed via an RPC, rather than inserting with `status: 'completed'` directly (which RLS would reject).
5. **Tickets never persisted at all** — unlike the Firebase-Auth-guard pattern above, `createTicket`/`replyToTicket` (the user-facing ones) weren't guarded into a no-op — they simply never wrote to Firestore in the first place, only `setUser(...)`. A created ticket or reply lived in React state only and vanished on refresh. 18. **`claim_investment_payout` had no maturity check — an instant-profit exploit.** Found by auditing the deployed RPC before wiring the claim button (bug #17's sub-bullet). 19. **`create_copy_trade` trusted client-computed financial terms — arbitrary-payout exploit.** (Found in the July 2026 dedicated RPC security audit prompted by bug #18.) The RPC accepted `p_roi_percent`, `p_expected_profit`, `p_total_return`, `p_start_timestamp`, `p_end_timestamp`, `p_remaining_days` from the client and **inserted them verbatim** into `copy_trades`, validating only `balance >= amount_invested`. ### Investment top-up behavior (documented, intentional — do NOT "fix")

Verified in the July 2026 audit against a real topped-up row. When a user tops up a running investment, the `top_up_investment` RPC recomputes against the **blended** amount: `expected_profit = new_total_amount × roi_percent/100`, `total_return = amount + expected_profit` (e.g. a $20 top-up on a $1,000 Silver position → `amount 1020`, `expected_profit 183.60`, `total_return 1203.60` — internally consistent and correct). **Deliberate product behaviors, not bugs:** (1) the added funds earn the **full-term** ROI% (no pro-rating for a top-up that happens partway through the term); (2) a top-up does **not** extend `end_date` or reset `start_date` — the blended amount matures on the original schedule. This is generous-to-the-user and consistent; leave it as-is unless product explicitly changes the rule.
